## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → process | `token.json` / `credentials.json` (OAuth secrets) read from disk into the process |
| config (.env) → process | Sender list and lookback window read via `os.getenv`; OAuth scope MUST NOT cross this boundary |
| package registry → environment | `pip install` of the Google stack + stubs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-scope | Tampering | OAuth scope in `auth.py` | mitigate | Hard-code `SCOPES` constant to `gmail.readonly`; never read scope from `.env`. Acceptance criterion asserts the exact constant value and that scope is not from `os.getenv`. |
| T-02-token | Info Disclosure | `token.json` / `credentials.json` | mitigate | Both git-ignored (Phase 1 T-01-01); `load_credentials()` never logs the Credentials object or any field. README (Phase 6) documents `chmod 600 token.json`. |
| T-02-logpii | Info Disclosure | logging in `auth.py` | mitigate | `auth.py` logs no credential fields; only opaque messages. (Body/sender PII logging is covered in 02-02 where the fetch loop lives.) |
| T-02-SC | Tampering | pip installs (google stack + stubs) | accept | All four packages passed the RESEARCH §Package Legitimacy Audit slopcheck ([OK], none `[ASSUMED]`/`[SUS]`/`[SLOP]`); established Google-published packages + verified stubs (henribru). No blocking checkpoint required. |
</threat_model>
