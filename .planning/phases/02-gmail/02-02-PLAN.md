<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Gmail API ŌåÆ process | Untrusted message payloads (headers, base64url bodies) enter the process via `messages.get` |
| process ŌåÆ log file | Email content must NOT cross into logs (PII boundary) |
| config (.env) ŌåÆ process | `GMAIL_SENDER_LIST` / `GMAIL_LOOKBACK_DAYS` read via `os.getenv` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-logpii | Info Disclosure | logging in `client.py` fetch loop | mitigate | Only `message_id=`/`count=` logged; LOG-02 `caplog` test asserts no `@` and no body token in log output. |
| T-02-input | Denial of Service / Tampering | `build_query` inputs + base64url decode | mitigate | `window_days` cast to int from env; `_decode_base64url` normalises padding and decodes with `errors="replace"` so malformed bodies cannot raise/crash the run. |
| T-02-scope | Tampering | readonly fetch | mitigate | Fetch only calls `messages.list`/`messages.get` (read ops); never `modify`/`trash`/`delete`. Scope constant enforced in 02-01 `auth.py`. |
| T-02-quota | Denial of Service | `messages.get` rate limit (429/403) | accept | `newer_than:30d` + sender-scoped query bounds volume for a personal mailbox; `_execute_with_backoff` handles transient 429/403. Quota exhaustion on huge mailboxes is a documented operational limit (RESEARCH Pitfall 6), not a code defect. |
</threat_model>
