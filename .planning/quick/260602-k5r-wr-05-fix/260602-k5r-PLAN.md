<objective>
Fix WR-05 (Phase 05 review): the registrar's retry pauses are an unbounded,
un-jittered fixed sleep. On a timeout/connect error or 5xx, __call__ does
time.sleep(self._retry_pause) (default 2.0s). Each __call__ retries at most once,
but the TrackingMoreRegistrar instance is reused for every email in a run, so a
batch of N transient-failing emails accumulates up to N×2s of serial sleep with no
upper bound — risking collision with the cron interval and overlapping runs. There
is also no jitter, so retries against a recovering API can synchronize.

DECISION (see 260602-k5r-CONTEXT.md): Budget + jitter.
- Bound total retries across the run with a per-instance budget.
- Add random jitter to each retry pause (like the Gmail client's
  _execute_with_backoff random.uniform(0,1)).

Purpose: bound worst-case cumulative retry latency per run and de-synchronize
retries.
Output: bounded+jittered retry in registrar.py + regression tests.
</objective>

       max_total_retries: int = 8,
   Store self._retry_jitter = retry_jitter and self._retry_budget = max_total_retries.
   Document (LOG-02-safe) that the budget is per-run because the instance is reused
   across every email in a cron run.
3. Add a private helper that consumes one unit of budget and sleeps with jitter,
   returning whether a retry may proceed:

       def _sleep_for_retry(self) -> bool:
           if self._retry_budget <= 0:
               return False
           self._retry_budget -= 1
           time.sleep(self._retry_pause + random.uniform(0, self._retry_jitter))
           return True

4. Replace BOTH retry sites in __call__:
       if attempt == 0:
           time.sleep(self._retry_pause)
           continue
       return False
   with:
       if attempt == 0 and self._sleep_for_retry():
           continue
       return False
   (Applies to both the (TimeoutException, ConnectError) branch and the
   HTTPStatusError branch.) This preserves the at-most-one-retry-per-call behavior
   while the budget caps cumulative retries across the batch; when the budget is
   exhausted, even attempt 0 defers (returns False).
