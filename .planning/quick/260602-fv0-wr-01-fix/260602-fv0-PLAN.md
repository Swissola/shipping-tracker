<objective>
Fix WR-01: the docstring on `register_and_persist` (shipping_tracker/db.py:74-90)
falsely claims the function is "Idempotent / safe to retry" and that a repeat call
for an already-present tracking_number "is a silent no-op ... and still returns True."

This is wrong at the cost layer. The function calls `registrar(tracking_number, carrier)`
unconditionally at db.py:92 — a billable TrackingMore API call — before any DB lookup.
Only the DB writes are idempotent (INSERT OR IGNORE). A caller trusting the docstring and
invoking the function directly for an already-registered number would burn free-tier quota.

Per the locked decision, this codebase deliberately centralizes dedup policy in the
orchestrator (main.py DEDUP-03/04/05). `register_and_persist` is a lower-level primitive:
"call registrar, persist atomically on success." Adding an internal guard would duplicate
policy and be dead code in the real pipeline (main.py:136 always pre-empts it). The defect
is purely the lying docstring.

Purpose: Make the documented contract match actual behavior so future callers do not waste
billable API quota by trusting a false idempotency promise.
Output: Corrected docstring on register_and_persist. Zero executable-code changes.
</objective>
