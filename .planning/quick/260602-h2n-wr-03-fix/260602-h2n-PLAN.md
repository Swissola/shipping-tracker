<objective>
Fix WR-03 (Phase 05 review): configure_logging (shipping_tracker/logging_config.py:48-50)
calls root_logger.addHandler(handler) without removing existing handlers and
unconditionally sets the root level. Each call to main() — and each test that invokes
main() — appends another RotatingFileHandler and opens another file handle on
logs/shipping-tracker.log. Within one process that calls main() more than once,
handlers and open file descriptors accumulate and log lines are emitted multiple times.

Fix: make configure_logging idempotent — remove and close existing root handlers
before adding the new one.

Purpose: Eliminate handler/fd leak and duplicate log emission; remove the test-suite hazard.
Output: Idempotent configure_logging + a regression test proving no accumulation.
</objective>

  <action>
In shipping_tracker/logging_config.py, replace the root-handler setup (currently
lines 48-50):

    root_logger = logging.getLogger()
    root_logger.addHandler(handler)
    root_logger.setLevel(log_level)

with an idempotent form that clears existing handlers first:

    root_logger = logging.getLogger()
    for existing in list(root_logger.handlers):
        root_logger.removeHandler(existing)
        existing.close()
    root_logger.addHandler(handler)
    root_logger.setLevel(log_level)

Keep the structlog.configure(...) block and the "NO StreamHandler" comment unchanged.

Add tests/test_logging_config.py with a regression test that:
- Saves the current root_logger.handlers and level, and restores them in a finally
  block (so the test does not pollute the rest of the suite — important because the

- Points log_path at tmp_path (e.g. tmp_path/"logs"/"t.log") so nothing is written
  under the repo logs/ dir.
- Calls configure_logging(log_path=..., log_level=logging.WARNING) twice.
- Asserts the count of RotatingFileHandler instances on the root logger is exactly 1
  after the second call (proving no accumulation), and that the root level is WARNING.
